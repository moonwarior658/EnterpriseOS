# EnterpriseOS — Project Charter

> Версия: 1.1
> Статус: Активная разработка
> Последнее обновление: 18 июля 2026

---

# 1. Миссия проекта

EnterpriseOS — это единая операционная система компании.

Ее задача — объединить все бизнес-процессы компании в одном месте, убрать ручную рутину, сократить количество используемых программ и предоставить сотрудникам единое рабочее пространство.

Мы не строим ERP.

Мы строим операционную систему предприятия.

---

# 2. Цель проекта

Главная цель EnterpriseOS — освободить рабочее время сотрудников за счет автоматизации повторяющихся операций.

Критерий успеха:

- сотрудники тратят меньше времени на поиск информации;
- сотрудники не дублируют ввод данных;
- сотрудники работают в одном интерфейсе;
- руководители видят только то, что требует их участия.

Конечная цель:

Компания работает как единая система, а не как набор отдельных программ и чатов.

---

# 3. О компании

Компания занимается производством и продажей кондитерских изделий, выпечки и готового питания.

В составе компании:

- Производственный цех
- Склад
- Снабжение
- Логистика
- Торговые точки
- Кафе
- Техническая служба
- Бухгалтерия
- Руководство

Компания развивается быстрее существующих процессов, поэтому основной задачей EnterpriseOS является масштабирование управления.

---

# 4. Основной принцип

Система должна подстраиваться под компанию.

Компания не должна менять процессы ради системы.

---

# 5. Философия разработки

Мы не проектируем идеальную систему.

Мы ежедневно улучшаем существующую.

Каждый рабочий день должен заканчиваться результатом.

Результатом считается:

- новый модуль;
- новый API;
- исправленная ошибка;
- рабочий экран;
- новый документ;
- автоматизация существующего процесса.

---

# 6. Основные принципы

1. Простота важнее красоты.
2. Понятность важнее сложности.
3. Рабочий код важнее обсуждений.
4. Минимум ручной работы.
5. Один источник истины для данных.
6. Каждая автоматизация должна экономить время.
7. Архитектура должна допускать развитие без переписывания проекта.
8. Пользователь управляет бизнесом через EnterpriseOS, а не через технические инструменты.
9. Базовая настройка типового процесса должна занимать не более четырех понятных действий.
10. Внешние сервисы и оркестраторы скрыты за единым интерфейсом EnterpriseOS.

---

# 7. Что EnterpriseOS НЕ делает

EnterpriseOS не пытается заменить физическую работу.

Она автоматизирует все действия вокруг нее.

Например:

Нельзя автоматизировать:

- ремонт оборудования;
- инвентаризацию;
- приемку товара;
- приготовление продукции.

Можно автоматизировать:

- постановку задач;
- документы;
- уведомления;
- учет;
- контроль;
- аналитику.

# 8. Интерфейс

Основной ориентир — философия macOS.

Минимализм.

Отсутствие информационного шума.

# 9. Архитектура

EnterpriseOS состоит из независимых модулей.

Например:

Core

Supply

Warehouse

Production

Logistics

ServiceDesk

KnowledgeBase

Documents

Analytics

AI

Каждый модуль развивается независимо.

EnterpriseOS является управляющим ядром и единственным источником истины для бизнес-данных.

Система разделяется на два контура:

## Управляющий контур EnterpriseOS

- пользователи, компании и права доступа;
- справочники и бизнес-данные;
- регламенты, задачи и расписания;
- бизнес-правила и расчеты;
- история, аудит и аналитика;
- пользовательский интерфейс;
- внутренние коммуникации.

## Исполнительный контур автоматизации

Исполнительный контур выполняет технические действия:

- обращения к внешним API;
- отправку сообщений и уведомлений;
- запуск многошаговых интеграционных процессов;
- повторные попытки после временных ошибок;
- прием webhook-событий;
- возврат результата в EnterpriseOS.

Первым исполнителем автоматизаций является n8n.

n8n подключается через заменяемый интерфейс `AutomationProvider` и не является источником истины.

Обязательные архитектурные правила:

1. n8n не обращается напрямую к базе данных EnterpriseOS.
2. n8n не хранит бизнес-состояние компании.
3. Критическая бизнес-логика и расчеты не размещаются в Code Node.
4. EnterpriseOS и n8n взаимодействуют только через версионируемые API и webhook-контракты.
5. Каждый запуск имеет уникальный идентификатор и защиту от повторного выполнения.
6. Результат каждого запуска возвращается и сохраняется в EnterpriseOS.
7. Пользовательские регламенты не создают отдельный workflow для каждого пользователя.
8. В n8n используются универсальные workflow-шаблоны.
9. Обычные пользователи и администраторы не работают с интерфейсом n8n.
10. Замена n8n не должна требовать переписывания бизнес-модулей EnterpriseOS.

# 14. Видение проекта

Через несколько лет сотрудник открывает только EnterpriseOS.

Все остальные системы продолжают работать, но остаются скрытыми за единым интерфейсом.

EnterpriseOS становится единой точкой входа во всю деятельность компании.

В долгосрочной перспективе внутреннее общение компании также переносится в EnterpriseOS.

Сотрудники получают личные сообщения, рабочие каналы, файлы, задачи, фотоотчеты и уведомления внутри единой системы.

Telegram и другие мессенджеры используются как переходные или внешние каналы, но не остаются главным местом хранения рабочих договоренностей.

Сообщение внутри EnterpriseOS может быть связано с задачей, заявкой, документом, оборудованием, поставкой или другим объектом системы.

# 16. Принцип пользовательского интерфейса

EnterpriseOS создается для обычных сотрудников и администраторов, которые не обязаны понимать код, API, n8n или устройство автоматизаций.

Пользователь видит:

- что произойдет;
- когда это произойдет;
- кто отвечает за результат;
- последний результат;
- следующую дату запуска;
- понятное описание ошибки.

Технические настройки доступны только разработчикам и роли `platform_admin`.

Типовой процесс настраивается через шаблон:

1. Выбрать действие.
2. Выбрать подразделение или объект.
3. Указать расписание и получателей.
4. Нажать «Включить».

# 17. Коммерческая модель автоматизации

EnterpriseOS проектируется как самостоятельный коммерческий продукт.

Для внутренней эксплуатации компании допускается собственный self-hosted экземпляр n8n Community Edition.

При поставке EnterpriseOS другой компании основной вариант развертывания следующий:

- клиент самостоятельно владеет своим экземпляром n8n;
- n8n работает на инфраструктуре клиента и используется для его внутренних процессов;
- учетная запись владельца, база данных, ключ шифрования и credentials принадлежат клиенту;
- EnterpriseOS подключается к n8n через `N8nProvider`;
- n8n юридически и технически остается отдельным сторонним компонентом;
- EnterpriseOS не продает доступ к n8n и не выдает n8n за собственный продукт;
- перед первой коммерческой поставкой схема лицензирования получает письменное подтверждение правообладателя и юридическую проверку.

Архитектура EnterpriseOS не должна зависеть от бесплатной лицензии конкретного оркестратора.
