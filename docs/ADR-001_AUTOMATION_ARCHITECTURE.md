# ADR-001 — Архитектура автоматизаций EnterpriseOS

Статус: Принято

Дата: 18 июля 2026

## Контекст

EnterpriseOS должна автоматизировать снабжение, склад, техническую поддержку, логистику, работу с подрядчиками, уведомления и другие повторяющиеся процессы.

В пре-альфа концепции n8n выполнял значительную часть расчетов и управления процессами.

Такой подход удобен для быстрого прототипа, но создает риски:

- бизнес-логика скрывается внутри workflow;
- данные и состояние распределяются между системами;
- процессы сложно тестировать и версионировать;
- пользовательский интерфейс зависит от технического инструмента;
- замена оркестратора требует переписывания продукта;
- коммерческая поставка зависит от лицензии n8n.

Обычные пользователи и администраторы EnterpriseOS не обладают техническими знаниями и не должны работать с кодом, API или редактором n8n.

## Решение

EnterpriseOS является управляющим ядром и единственным источником истины.

n8n используется как скрытый исполнительный оркестратор и подключается через интерфейс `AutomationProvider`.

Первая реализация интерфейса называется `N8nProvider`.

## Ответственность EnterpriseOS

EnterpriseOS хранит и обрабатывает:

- пользователей, компании, подразделения и права;
- регламенты и расписания;
- задачи и экземпляры процессов;
- бизнес-правила;
- расчеты и рекомендации;
- статусы и историю;
- аудит;
- пользовательский интерфейс;
- понятные сообщения об ошибках.

## Ответственность n8n

n8n выполняет:

- обращения к внешним API;
- Telegram- и email-уведомления;
- прием webhook-событий;
- многошаговые интеграционные процессы;
- ожидание внешних событий;
- повторные попытки технических операций;
- возврат результата в EnterpriseOS.

## Топология внутренней эксплуатации

Целевая production-топология требует локального размещения EnterpriseOS и предназначенного для него экземпляра n8n на основном сервере компании.

`N8nProvider` не должен зависеть от NL VPS.

Существующий NL VPS и размещенный на нем n8n остаются временным legacy-контуром для Telegram-бота, прокси и уже работающих процессов. После переноса внутренних чатов и процессов в EnterpriseOS NL VPS планируется отключить.

Локальный n8n взаимодействует с EnterpriseOS только через API и webhook и не получает прямого доступа к PostgreSQL.

## Запрещенные решения

Запрещается:

1. Давать n8n прямой доступ к базе EnterpriseOS.
2. Хранить окончательное бизнес-состояние только в n8n.
3. Размещать критические расчеты в Code Node.
4. Создавать отдельный workflow для каждого пользовательского регламента.
5. Показывать интерфейс n8n обычным пользователям и администраторам.
6. Использовать внутренний идентификатор workflow как бизнес-идентификатор EnterpriseOS.
7. Считать успешный HTTP-запрос доказательством успешного завершения бизнес-процесса.
8. Связывать бизнес-модули напрямую с API конкретного оркестратора.

## Контракт выполнения

Каждая команда автоматизации содержит минимум:

- `contract_version`;
- `execution_id`;
- `automation_type`;
- `tenant_id`;
- `requested_at`;
- `payload`;
- `callback_url`.

Каждый результат содержит минимум:

- `contract_version`;
- `execution_id`;
- `status`;
- `started_at`;
- `finished_at`;
- `result`;
- `error_code`;
- `error_message`.

Повторная доставка команды с тем же `execution_id` не должна повторно создавать бизнес-операцию.

Исходящие команды публикуются через transactional outbox.

## Пользовательский интерфейс

Пользователь создает автоматизацию через шаблон EnterpriseOS:

1. Выбирает процесс.
2. Выбирает область действия.
3. Указывает расписание и получателей.
4. Нажимает «Включить».

Пользователь видит результат, следующий запуск и понятное описание проблемы.

Техническая диагностика доступна только разработчикам и роли `platform_admin`.

## Коммерческое развертывание

Для собственной компании используется собственный self-hosted экземпляр n8n.

Основная модель поставки другому клиенту:

- клиент владеет своим экземпляром n8n;
- n8n работает на инфраструктуре клиента;
- credentials и ключи принадлежат клиенту;
- EnterpriseOS подключается через `N8nProvider`;
- n8n остается отдельным сторонним компонентом;
- перед первой продажей модель получает письменное подтверждение правообладателя и юридическую проверку.

## Последствия

Положительные последствия:

- пользователи не зависят от сложности n8n;
- бизнес-логика тестируется обычными средствами разработки;
- данные и история остаются в EnterpriseOS;
- n8n можно обновлять или заменять независимо;
- каждый клиент может использовать собственный оркестратор;
- снижается лицензионная зависимость продукта.

Цена решения:

- необходимо разработать Automation Core;
- требуется надежный обмен событиями;
- нужны idempotency, outbox, callback и мониторинг;
- часть логики, которую можно быстро написать в n8n, будет реализована в backend EnterpriseOS.

Эта цена принимается, потому что она предотвращает более дорогую зависимость и переписывание продукта в будущем.

## Критерий проверки решения

Первый сквозной процесс считается успешным, если:

1. Пользователь создает регламент в EnterpriseOS.
2. EnterpriseOS создает запуск и `execution_id`.
3. `N8nProvider` передает команду в универсальный workflow.
4. n8n выполняет внешнее действие.
5. n8n возвращает callback.
6. EnterpriseOS сохраняет результат и показывает его пользователю.
7. Повторная доставка той же команды не создает повторное действие.
8. Отключение n8n приводит к понятному статусу, а не к потере данных.
